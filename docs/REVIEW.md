## 3. 리스크 검토

### 높은 우선순위 리스크

#### R1. AI 파싱 전략 정확도
- **문제**: Claude가 생성한 JSON 전략이 실제 크롤링에서 동작하지 않을 수 있음
- **징후**: CSS 선택자 오류, 잘못된 pagination 타입
- **완화**:
  - 전략 JSON 스키마 검증 레이어 구현 (S4-1)
  - 관리자 UI에서 JSON 수동 수정 기능 제공 (S3-2)
  - 실패 시 자동 재분석 트리거 (S4-4)
  - 프롬프트에 파싱 전략 예시 few-shot 포함

#### R2. 크롤링 대상 사이트 구조 변경
- **문제**: 게임 공식 사이트가 리뉴얼되면 기존 전략 무효화
- **완화**:
  - 크롤링 실패 → FAILED 상태 자동 전환
  - 관리자 UI에서 재분석 버튼으로 즉시 대응
  - 기존 전략 버전 보관 (rollback 가능)

#### R3. Claude API 비용
- **문제**: 대량 이미지 태그 생성/설명 생성 시 비용 급증
- **추정**: 이미지당 $0.01~0.03 × 대량 처리
- **완화**:
  - 태그 생성은 크롤링과 별도 비동기 처리 (속도 제어 가능)
  - 배치 처리 시 rate limit 준수 (분당 호출 수 제한)
  - 프롬프트 토큰 최적화 (HTML 불필요 태그 제거)
  - 재분석 자동 트리거 횟수 제한

### 중간 우선순위 리스크

#### R4. Selenium 메모리 누수
- **완화**: Docker `shm_size: 2g`, 크롤링 완료 후 세션 명시적 종료, 컨테이너 자동 재시작 설정

#### R5. SQLite 동시성 제한
- **완화**: WAL 모드 활성화, 쓰기 직렬화 (Phase 1 범위에서는 단일 서버이므로 낮은 리스크)

## 4. Sprint 간 의존성 검토

```
S1 (인프라) → S2 (크롤러 마이그레이션) 의존
S3 (Admin UI 기초 + Claude API) → S4 (AI 크롤러 핵심) 의존
S4 (GenericCrawler) → S5 (태그/추천) 의존
S5 (태그/추천) → S6 (리팩토링, 테스트) 독립 진행 가능
S6 (리팩토링) → S7 (데스크탑) 독립 진행 가능
S7 → S8 순차 진행
```

**주의**: S3에서 Claude API 기초 연동을 완료해야 S4 AI 파싱 전략 생성이 가능. S3 지연 시 S4 일정에 직접 영향.
